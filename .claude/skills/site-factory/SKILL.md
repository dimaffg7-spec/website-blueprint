# site-factory - конвейер «сайт под ключ»

Оркестратор. Ведёт проект дисциплинированно, не пропуская слои. Полная методология - в репозитории **website-blueprint** (`docs/`, `prompts/`, `templates/`). Если репозиторий доступен в проекте или рядом - читай оттуда детали; если нет - следуй краткой версии ниже.

## Шаг 0 - классификация

Спроси, если неясно:
1. **Тип:** сайт заказчику с нуля (полный конвейер) или доработка живого сайта (режим волн)?
2. **Стек:** Next.js / Astro / чистый HTML / другой?
3. **Стиль:** есть бренд-бук / референсы, или собираем дизайн-систему с нуля?

## Шаг 1 - День 0: бриф БЕЗ КОДА

**Первый вопрос - хостинг** (главный урок методологии: среда исполнения выясняется до выбора архитектуры; форма под Node на shared-хостинге с PHP = день переделки). Задай бриф-вопросы:
- хостинг и среда (PHP/Node/статика? доступ к панели и DNS?);
- контент (текущий сайт? тексты 1:1?), материалы (фото, лого, регалии, реальные отзывы);
- юр (реквизиты, оферта, политика ПДн, лицензии; собираем ли ПДн);
- куда падают заявки; SEO-наследство (карта 301); референсы вкуса.

Не пиши код, пока нет ответов. Разложи в `business/` (raw, brand, products, audience, seo) + `plans/`.

## Шаг 2 - План

Создай `plans/YYYY-MM-DD-site-<имя>.md`: фазы 0 каркас → 1 дизайн-система (токены + `/styleguide`) → 2-3 страницы → 4 форма (под реальный хостинг!) → 5 SEO → 6 аналитика → 7 юр → 8 GEO → 9 финал. **У каждой фазы - готовый промпт запуска.** Прогони **«10 причин провала»** ДО кода → блок рисков в план.

## Шаг 3 - Ведение по слоям

На каждом слое держи чеклист:
- **Дизайн:** токены + `/styleguide`-эталон; стиль из письменного бренд-спека (нет - собери из референсов, «извлеки систему, не копируй пиксели»); тёмное = акцент-сцена, не фон всего сайта; референсы портировать на CSS, не тащить рантаймы; reveal-on-scroll с гейтом по `.js`; каждой анимации - `prefers-reduced-motion`.
- **SEO:** уникальные title/description под кластер; canonical; og-image PNG 1200×630; sitemap/robots; allow-лист ИИ-краулеров (GPTBot, ClaudeBot, PerplexityBot, Google-Extended, YandexBot и др.); verification (Google + Яндекс).
- **GEO:** JSON-LD граф через `@id` (Person↔Org↔WebSite↔Course/Product↔FAQ↔Breadcrumb); FAQPage из видимого FAQ; Review только на Course/Product из реальных видимых отзывов; llms.txt + llms-full.txt; answer-first; факт-плотность; дата обновления.
- **Performance:** Lighthouse ≥95 (гейт фазы); keyframe-видео для скраба + poster; баны `scale`-на-blur и `backdrop-blur`-на-движущемся; гейт бесконечных анимаций; WebP + lazy; self-host шрифтов.
- **Форма:** 3 канала на заявку (мессенджер+email+лог); honeypot+rate-limit; согласие ПДн; секреты вне git; ПДн-логи над веб-корнем.
- **Аналитика:** цепочка согласие→счётчик→цель, одно хранилище.
- **Юр:** дословно из документов, ничего не выдумывать.
- **Конверсия:** аудит отдельным свежим чатом - «5 линз (StoryBrand + Direct Response + ясность + потери по экранам + копирайтинг) → 2 проверки (фактчек + бренд) → синтез»; каждый слабый заголовок - 3 варианта.

**Режимы чатов:** большой чат = сборка фаз; полировка = «одна правка = один чат» + коммит с человеческим именем; аудиты = отдельный свежий чат.

## Шаг 4 - Финал

Pre-flight (билд зелёный, Lighthouse ≥95, тестовая заявка насквозь, цепочка аналитики) → security review → деплой (только по явной команде, после зелёного билда; `git push` = бэкап, не деплой) → sitemap в вебмастеры (регион) → передача: хвосты владельцу + ретроспектива. **Новые грабли дописать в методичку** - она живая.

## Жёсткие правила

- **Не выдумывать:** цифры, отзывы, лица, юр-формулировки - только подтверждённые. Нет данных → `[уточнить]`.
- **Деплой только по явной команде** владельца, после зелёного билда.
- В git добавлять поимённо; секреты (`.env`, токены) не читать целиком и не светить.
- Большая фича без плана не начинается; план актуализировать после каждой сессии.
