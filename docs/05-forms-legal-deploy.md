# 05 - Форма, юр-обвязка, деплой

Инфраструктурный слой: как принять заявку, не нарушить закон о данных и выкатить сайт.

## Форма и лиды

- **Обработчик под реальный хостинг** (определяется в брифе, день 0):
  - shared без Node → статика + серверный скрипт рядом (например PHP-обработчик);
  - Node/SSR → API-эндпоинт фреймворка.
- Проверенный минимум обработчика: валидация, honeypot (ловушка для ботов), rate-limit, отправка в мессенджер + email, лог в файл. **Секреты (токен бота) - в отдельном конфиге вне git.**
- **3 канала на заявку: мессенджер + email + лог** - лид не должен пропасть ни при каком одиночном отказе.
- Чекбокс согласия на обработку ПДн + ссылки на юр-страницы прямо у формы.
- Тестовая заявка проходит насквозь до передачи.
- **Серверные файлы на прод сами не попадают** при статической сборке - нужен отдельный заливка/чек-лист «что на проде».

## Юр-обвязка (ориентир для РФ, не юр-консультация)

- Оферта, политика обработки ПДн, согласия, сведения об организации - **дословно** из документов заказчика. Не редактировать самодеятельно.
- Числовые обещания (рассрочка, гарантии) - только подтверждённые формулировки. Пример: без расчётных сумм «от N в месяц», если условия зависят от третьих сторон - только подтверждённый максимум.
- Лицензии - кликабельны, ведут на официальные выписки.
- Из футера убрать домашний адрес физлица-владельца (это тоже ПДн).
- **Логи заявок содержат ПДн** → хранить над веб-корнем (недоступно по прямому URL) + закрыть в robots.
- Если сайт собирает ПДн - уведомление регулятора в список хвостов заказчику.

## Деплой

Для статического сайта на shared-хостинге - два проверенных пути:

**Путь А - FTP-скрипт.** Сборка статики → заливка по FTPS скриптом (можно на голой stdlib, без зависимостей). Безопасный порядок флагов:
```
1. --list        # убедиться, что попал в нужную папку (не в соседний сайт на общем логине)
2. --backup DIR  # скачать текущий живой сайт (откат)
3. --dry-run     # план заливки, ничего не трогая
4. --clean       # удалить старьё - ТОЛЬКО после бэкапа; защитить системные папки хостинга
5. (без флага)   # залить
```
Креды - в env-файле вне git (не читать целиком, не светить пароль в логах). После заливки - проверка live через `curl`.

**Путь Б - архив через панель хостинга.** Собрать → zip → залить через файловый менеджер панели.

Правила деплоя:
- Деплой - **только по явной команде владельца** и после зелёного билда (полная сборка, не только проверка типов; exit code проверять напрямую, не через пайп в `tail` - он маскирует провал).
- **`git push` в бэкап-репозиторий - это НЕ деплой.** Выкат прода - отдельный канал (FTP/панель). Не путать.
- Перед сборкой остановить dev/preview-сервер (гонка за общий build-кэш роняет сборку при статическом экспорте).

### Осторожно с серверным конфигом
Файл вроде `.htaccess` может **уронить сайт** на старом Apache (несовместимый синтаксис директив). Проверить версию сервера прежде чем полагаться на него; при сомнении собирать архив без него, а редиректы/кэш решать иначе.
